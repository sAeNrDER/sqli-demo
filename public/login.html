🧪 ACTIVITY APPRENTICE

Retrieving Hidden Data

Imagine a shopping application that displays products in different categories. When the user clicks the Books category, their browser requests the URL:

https://insecure-website.com/products?category=Books

This causes the application to make a SQL query to retrieve the relevant products from the database:

SELECT * FROM products WHERE category='Books' AND released=1

This SQL query asks the database to return:

  • all details (*)
  • from the products table
  • where the category is Books
  • and released is 1

The restriction released=1 is being used to hide products that are not released. We could assume for unreleased products, released=0. The application doesn't implement any defenses against SQL injection, so an attacker can construct the following attack:

https://insecure-website.com/products?category=Books'--

This results in the SQL query:

SELECT * FROM products WHERE category='Books'-- AND released=1
  Crucially, -- is a SQL comment. The rest of the query is removed.
  released=1 no longer applies — ALL Books products are displayed.

You can use a similar attack to cause the application to display all products in any category, including categories it doesn't know about:

https://insecure-website.com/products?category=Gifts'+OR+1=1-- 
SELECT * FROM products WHERE category='Gifts' OR 1=1-- AND released=1
  The modified query returns ALL items where category='Gifts' OR 1=1.
  Since 1=1 is always true, the query returns everything.
Warning: Take care when injecting OR 1=1. Even if it appears harmless in a SELECT context, it's common for applications to reuse data from one request in multiple queries. If your condition reaches an UPDATE or DELETE statement, it can result in accidental loss of all data.

Vulnerable vs. Safe — WHERE Clause

// GET /api/products?category=Books
const sql = `SELECT * FROM products
  WHERE category='${category}'
  AND released=1`;

// Attacker injects:  Books'--
// SQL becomes:
// WHERE category='Books'-- AND released=1
// released=1 check is GONE!
✅ Safe (parameterized)
// Input treated as pure data, never SQL
const sql = `SELECT * FROM products
  WHERE category=?
  AND released=1`;

db.prepare(sql).all(category);

// Injecting Books'-- just looks for a
// category literally named "Books'--"
// No rows match → empty result, no bypass
🛡️ PREVENTION

How to Prevent SQL Injection

Most SQL injection vulnerabilities can be prevented by using parameterized queries (also called prepared statements) instead of string concatenation. The query structure is defined as a hard-coded template — user input is passed separately and the database never confuses it for SQL syntax.

✨ Golden Rule

The query template must be a hard-coded constant. No part of the SQL structure should ever come from user input — no matter how trusted the source appears.

Step-by-step: Fixing a Vulnerable Query

1

Identify string concatenation

Any place where user input is embedded into a SQL string using +, template literals, %s, or format() is potentially vulnerable.

// ❌ BAD — user input inside the string
const sql = `SELECT * FROM users WHERE username='${username}'`;
2

Replace with a placeholder

Replace the user value with a ? placeholder (or $1 in PostgreSQL). The SQL structure is now fixed and cannot be changed by any input.

// ✅ GOOD — structure is fixed, ? is a placeholder
const sql = `SELECT * FROM users WHERE username=?`;
3

Pass input as a separate parameter

Provide the user input to the database driver separately. The driver handles escaping and ensures the value is always treated as data — never as SQL syntax.

// ✅ Input passed separately — can never escape the string context
db.prepare(sql).all(username);

Examples Across Languages

💀 Vulnerable — Node.js
// String interpolation = injectable
const sql = `SELECT * FROM users
  WHERE username='${username}'
  AND password='${password}'`;

db.prepare(sql).all();
✅ Safe — Node.js (better-sqlite3 / sql.js)
// ? placeholders — input is data only
const sql = `SELECT * FROM users
  WHERE username=?
  AND password=?`;

db.prepare(sql).all(username, password);
💀 Vulnerable — Python
# f-string inside SQL = injectable
cursor.execute(
  f"SELECT * FROM users WHERE username='{username}'"
)
✅ Safe — Python (sqlite3 / psycopg2)
# %s placeholder + tuple of params
cursor.execute(
  "SELECT * FROM users WHERE username=%s AND password=%s",
  (username, password)
)
💀 Vulnerable — Java (JDBC)
// Concatenation = injectable
String sql = "SELECT * FROM users WHERE "
  + "username='" + username + "' AND "
  + "password='" + password + "'";
Statement stmt = conn.createStatement();
ResultSet rs = stmt.executeQuery(sql);
✅ Safe — Java (PreparedStatement)
// ? placeholders — structure is fixed
PreparedStatement stmt = conn.prepareStatement(
  "SELECT * FROM users WHERE username=? AND password=?"
);
stmt.setString(1, username);
stmt.setString(2, password);
ResultSet rs = stmt.executeQuery();
💀 Vulnerable — PHP
// Variable in query string = injectable
$sql = "SELECT * FROM users WHERE
  username='$username' AND
  password='$password'";
$result = mysqli_query($conn, $sql);
✅ Safe — PHP (PDO)
// Named placeholders
$stmt = $pdo->prepare(
  "SELECT * FROM users WHERE username=:user AND password=:pass"
);
$stmt->execute([
  ':user' => $username,
  ':pass' => $password
]);

When Parameterized Queries Are Not Enough

Parameterized queries work for values (strings, numbers). They cannot be used for structural parts of a query like table names, column names, or ORDER BY values. For these, use an allowlist:

💀 Still vulnerable
// Column names can't be parameterized
const col = req.query.sort; // e.g. "username"
const sql = `SELECT * FROM users ORDER BY ${col}`;
// Attacker sends: sort=(SELECT password FROM users LIMIT 1)
✅ Safe — allowlist
// Validate against known-safe values only
const ALLOWED = ['username', 'email', 'created_at'];
const col = ALLOWED.includes(req.query.sort)
  ? req.query.sort
  : 'username'; // default fallback
const sql = `SELECT * FROM users ORDER BY ${col}`;
Summary of defences:
  • Parameterized queries — primary fix for all value inputs
  • Allowlist validation — for table/column names and ORDER BY
  • Least privilege — database user should only have SELECT where needed
  • WAF (Web Application Firewall) — useful as a secondary layer, not a replacement
