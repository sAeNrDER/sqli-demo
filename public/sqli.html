How SQL Injection Works

SQL injection (SQLi) is the #1 web vulnerability for over two decades. It happens when user-controlled input is embedded directly into a SQL query — letting attackers rewrite the query logic entirely.

1. What is SQL?

SQL (Structured Query Language) is used to talk to databases. Web apps run SQL queries constantly — fetching users, checking passwords, retrieving products.

-- Get all products in the 'electronics' category
SELECT name, price FROM products
WHERE category = 'electronics';

-- Check login credentials
SELECT * FROM users
WHERE username = 'admin' AND password = 'password123';

2. How the Vulnerability Arises

A developer writes code like this — taking input from a login form and embedding it directly into a query string:

// JavaScript / Node.js example
const username = req.body.username;  // e.g. "admin"
const password = req.body.password;  // e.g. "password123"

const sql = `SELECT * FROM users
             WHERE username='${username}'
             AND password='${password}'`;

With normal input, this produces a safe query:

SELECT * FROM users
WHERE username='admin' AND password='password123'

3. The Attack — Breaking Out of the String

An attacker enters ' or 1=1 -- as the username (note the trailing space after --). Let's trace what happens:

Input:
username = ' or 1=1 --
Password:
anything
Resulting SQL:
SELECT * FROM users WHERE username='' or 1=1 -- ' and password='anything'

Breaking it down piece by piece:

Fragment What it does
' Closes the opening quote around username — escapes the string context
or 1=1 Always evaluates to TRUE — makes the WHERE clause match every row
-- SQL comment — ignores everything after it (the password check disappears)
Result: The query returns ALL rows in the users table. The app sees "rows returned > 0" and logs you in as the first user — usually admin.

4. Error Probing — How Attackers Discover SQLi

Before crafting a bypass, attackers first confirm the field is injectable by sending a single quote ':

-- What gets built when username = '
SELECT * FROM users WHERE username=''' AND password='anything'
                                       ↑ Unmatched quote → syntax error!

The database throws a syntax error. If the app returns a 500 Internal Server Error (instead of just "invalid credentials"), it confirms the input is being passed to SQL — the field is injectable.

5. Other SQL Injection Types

1

WHERE clause injection (hidden data retrieval)

Used in product/search pages. Append ' OR '1'='1 to a category filter to retrieve hidden/unreleased items.

GET /products?category=Gifts' OR '1'='1

-- Becomes:
SELECT * FROM products WHERE category='Gifts' OR '1'='1'
2

UNION-based injection (data extraction)

Append a UNION SELECT to extract data from other tables — like usernames, passwords, or secrets.

' UNION SELECT username, password FROM users --
3

Blind injection (boolean/time-based)

When no error is shown, inject conditions that change page behaviour or add delays — inferring data one bit at a time.

' AND SLEEP(5) --   (page loads 5s slower = injectable!)

6. Defense — Parameterized Queries

The fix is simple: never concatenate user input into SQL. Use parameterized queries (also called prepared statements). The database treats the input as a literal value — not executable SQL.

💀 Vulnerable
// Input goes INTO the SQL string
const sql = `WHERE username='${u}'
             AND password='${p}'`;

// Attacker controls SQL logic!
✅ Safe (parameterized)
// ? placeholders — input is data only
const sql = `WHERE username=?
             AND password=?`;

db.prepare(sql).all(u, p);
// Input CANNOT change query structure
Rule of thumb: If you ever find yourself building a SQL string with +, template literals, or string formatting — stop and use a parameterized query instead.
