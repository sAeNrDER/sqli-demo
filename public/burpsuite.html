Burp Suite Guide

Burp Suite is the industry-standard web proxy used by security professionals. It sits between your browser and the server, letting you intercept, inspect, and modify every HTTP request before it's sent.

1. What is a Web Proxy?

Burp Suite sits in the middle — it can read, stop, and modify any request or response.

HTTP vs HTTPS: Standard HTTP is plaintext — easy to intercept. HTTPS is encrypted with TLS. Burp Suite handles HTTPS by acting as a man-in-the-middle: it generates its own certificates and its built-in browser is pre-configured to trust them, so it can decrypt and re-encrypt traffic transparently.

2. Intercepting HTTP Traffic (Lab §3.1)

1

Enable Interception

In Burp Suite: Proxy tab → Intercept sub-tab → toggle to Intercept on. All outgoing requests will now pause and wait for your approval.

2

Make a Request in Burp's Browser

Open the built-in browser (Proxy → Open browser) and navigate to a site. The page won't load — Burp has intercepted the request. You'll see the raw HTTP in the Intercept tab.

GET /burp HTTP/2 Host: portswigger.net User-Agent: Mozilla/5.0 (X11; Linux x86_64) ... Accept: text/html,application/xhtml+xml,*/* Cookie: session=abc123xyz...
3

Forward or Drop

Click Forward to send the request on to the server. Click Forward again for any follow-up requests until the page loads. Click Drop to discard the request entirely.

4

Turn Off Interception

Toggle back to Intercept off. Traffic will now flow through Burp without pausing — but it's still being logged in HTTP History.

3. Setting Target Scope (Lab §3.2)

Your browser sends requests to dozens of third-party hosts (ads, analytics, CDNs). The target scope filters these out so you only see traffic to the site you're testing.

1

Add Site to Scope

Go to Target → Site map. Right-click the target host in the left panel → Add to scope. When prompted, click Yes to stop logging out-of-scope traffic.

2

Filter HTTP History

In Proxy → HTTP history, click the filter bar at the top and enable Show only in-scope items. Now you only see requests to your target.

4. HTTP History (Lab §3.1 Step 5)

Even with Intercept off, every request passing through Burp is logged here. Click any row to see the full raw request on the left and the server response on the right. This lets you explore the site normally and review interactions afterwards.

HTTP History columns

5. Burp Repeater (Lab §3.3)

Repeater lets you re-send the same request as many times as you like, modifying parameters each time. This is essential for probing vulnerabilities without having to navigate the UI repeatedly.

1

Send to Repeater

In HTTP History, right-click any request → Send to Repeater (or Ctrl+R).

2

Modify and Resend

In the Repeater tab, edit any part of the request — headers, body, URL parameters — then click Send. The response appears on the right. Compare responses to discover vulnerabilities.

6. Performing SQL Injection with Burp Suite (Lab §4)

Here's the complete workflow for the PortSwigger login bypass lab:

1

Intercept the Login POST

Turn Intercept on. Submit the login form with any credentials. Burp catches the POST request:

POST /login HTTP/2 Host: 0a6f00fc04138618...web-security-academy.net Content-Type: application/x-www-form-urlencoded Content-Length: 69 csrf=abc123&username=some_user&password=some_password
2

Probe with a Single Quote

Change the username value to ' and click Forward. A 500 Internal Server Error confirms the field is injectable — the quote broke the SQL syntax.

csrf=abc123&username='&password=anything # Server responds: HTTP/2 500 Internal Server Error
3

Inject the Bypass Payload

Change the username to ' or 1=1 -- (note the space after --) and forward. The password check is bypassed and you're logged in as administrator.

csrf=abc123&username=' or 1=1 -- &password=anything # Server responds: HTTP/2 302 Found Location: /my-account?id=administrator
URL encoding: In the raw Burp request you'll see %27or+1%3D1+--+ — this is the URL-encoded form of 'or 1=1 -- . Burp handles the encoding automatically.
4

Use Repeater to Iterate Faster

Send the login POST to Repeater. Now you can try different payloads without enabling/disabling intercept each time. Ideal for testing variations like ' or '1'='1, ' or 1=1#, etc.
